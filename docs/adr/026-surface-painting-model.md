# ADR 026: Surface Painting Model (the Surface Brush)

Status: accepted
Date: 2026-07-13

## Context

Plan 068 set out to move asset appearance editing into the Layout
inspector and make it scopable. The real goal underneath, stated by
nikki, is a "super magic paint the world" experience: pick a surface
you already authored, arm a brush, paint on an asset, and the surface
appears where you painted -- with the option to open the actual UV,
layers, and masks to hand-tune. The technical detail (paint UVs,
masks, projection, inheritance) took a hard week of incremental
discovery to work out.

Before building the Surface Brush we researched how Substance Painter,
Blender, Unreal, and Unity do artist-friendly surface painting, and
mapped our own code against them. The conclusion: our data model had
already converged on the industry-standard architecture, and the
"magic" is an abstraction layer on top of it -- not a rewrite. These
decisions are settled; surface / painting / material work must follow
them, and future stories must not relitigate them without a new ADR.

## Decisions

### 1. The data model is the industry model -- keep it

A `Surface` is an ordered **layer stack**; each `Layer` carries a
`Mask`; masks are painted or procedural; a `SurfaceDefinition` is a
reusable library surface. This is exactly Substance Painter's
texture-set / layer / mask / smart-material model, Blender's layered
texture paint, and Unreal's masked material layers. The world-space
projection brush (paint in the 3D view, mark every texel of nearby
surface regardless of UV island) is Blender/Substance projection
painting. Do not redesign these primitives; build on them.

### 2. Surface is the shared "what"; coverage is the per-instance "where"

A painted look decomposes into two independently-owned parts:

- **The surface** (moss, grass, stone) -- the *what* -- is a SHARED,
  reusable library artifact (`SurfaceDefinition`, referenced by
  binding). Authored once, reused everywhere.
- **The coverage** (where that surface shows up on THIS object) --
  the *where* -- is a PER-INSTANCE painted mask on an INLINE surface
  override (`PlacedAssetInstance.surfaceSlotOverrides[].surface`, or
  the Scene tier).

The existing invariant `assertReusableSurfaceHasNoPaintedMasks`
(`domain/src/surface/index.ts`) enforces exactly this split: painted
masks are legal only on inline application-site surfaces, never on
library definitions. That is not a limitation to work around -- it is
the correct decomposition. You author "mossy stone" once; you paint
*where* the moss goes on each rock. The material is reused; only the
mask is per-object.

Consequence: painting a library surface onto an instance always
resolves that slot to an inline override first (fork the reference,
preserving its layers as the base -- `makeBindingLocal`), then adds a
masked layer carrying the chosen surface. The Surface Brush automates
this; it is not a new domain concept.

### 3. Masks live in TEXTURE space, not vertex colors

The mask substrate is a painted texture (`Mask` kind `painted` ->
`MaskTextureDefinition` -> `masks/*.png`), sampled through a
dedicated paint-UV channel (TEXCOORD_1 / `uv1`, authored in Blender
or generated by xatlas; see Plan 068.8).

Rejected alternative: vertex-color masks (`Mask` kind
`vertex-color-channel`, which exists and is consumed but is NOT the
painting substrate). Vertex colors are lighter and dodge the
paint-UV atlas entirely, but their resolution is capped by mesh
density. nikki's call (2026-07-13): the low-poly stylized meshes
cannot carry the *touches of hand-placed detail* that sell the
handcrafted painterly look (and match the established foliage style).
Texture masks are also already working. The vertex-color mask kind
remains available as a future per-surface lever, but the Surface
Brush paints texture masks.

Accepted cost: the paint-UV atlas is fragmented (a mesh can pack into
hundreds of small islands), which is why a texture-space circular
brush is WRONG -- it smears across unrelated islands or marks almost
nothing. Painting is therefore always WORLD-SPACE PROJECTION (walk
the hit mesh's triangles within a world radius, rasterize their
paint-UV footprints, value from real world distance). A texture-space
stamp is only correct for the landscape (a flat plane whose UV is
world XZ).

### 4. Paint in context; the Surface Studio is for tuning

The primary interaction is painting on the placed asset IN the Layout
viewport (in the real scene, with neighbors and lighting), not in an
isolated window -- matching how Substance and Blender paint in the
main 3D view. The **Surface Studio** (Plan 068.10) -- the layer stack
plus a UV view (mask + island wireframe) -- is opened for hand-tuning
what the brush created (reorder layers, swap the surface, adjust
masks, spot seams). A dedicated full-screen window is only warranted
if in-context proves cramped.

### 5. Scatter color inheritance is an advanced sub-feature, set down

Foliage scattered on a painted asset can inherit the asset's own
compiled-surface color (Plan 068.11: a top-down world-XZ surface bake
sampled by each blade's world XZ, the landscape ground-bake pattern
generalized). This works but is the heaviest, most fragile piece
(async mask load, build-order world bounds, top-down projection can't
see overhangs, CPU projection brush). It is DONE and SET DOWN: keep
it working, do not extend it (overhang correctness, higher fidelity)
until the core Surface Brush is proven in the field. It must never
block or complicate the core "paint a surface" path.

## Consequences

- The Surface Brush (Plan 068.9) is an abstraction over existing
  primitives: pick library surface -> resolve slot to inline ->
  auto-create a masked layer of that surface -> arm the projection
  painter. No new domain types.
- The manual flow (slot -> Inline Surface -> add layer -> set mask
  Painted -> paint) is demoted from primary to ADVANCED; the brush
  hides it, the Surface Studio (Plan 068.10) exposes it.
- Painting stays world-space projection for meshes, texture-circle
  for the flat landscape.
- Vertex-color painting is not built now; revisit only if texture
  masks prove insufficient in practice.
