# ADR 017: SugarDeploy Cloud Run Architecture

## Status

Accepted.

## Context

[Plan 021](/docs/plans/021-deployment-plugin-and-publish-deploy-target-architecture-epic.md)
established the deployment plugin architecture and named SugarDeploy as the
first-party deployment plugin. [Plan 045](/docs/plans/045-sugardeploy-cloud-run-plugin-owned-infrastructure-epic.md)
completes the SugarDeploy Cloud Run target end-to-end: provision, deploy,
secrets, lifecycle, drift handling, and major-version cuts. The decisions
below are the architectural rules the implementation settled on. They are
not aspirational; they are the rules new contributors and future agents
should read before changing the deploy plugin.

## Decision

### Plugin-owned terraform; `gcloud`-owned Cloud Run service

The plugin emits `main.tf`, `variables.tf`, `outputs.tf`, and
`terraform.tfvars` into `<game-root>/deployment/google-cloud-run/terraform/`
on every project save. Every file carries the
`GENERATED BY SUGARMAGIC - DO NOT EDIT` header and a
`# SUGARMAGIC TEMPLATE VERSION: NN` stamp. Terraform owns the slow-changing
surrounding infrastructure (Artifact Registry, runtime service account,
Workload Identity Federation pool/provider, IAM bindings, empty Secret
Manager containers). Terraform deliberately does NOT own the
`google_cloud_run_v2_service` resource. The Cloud Run service lifecycle is
`gcloud`-managed via the plugin-generated `deploy.sh` (create on first run,
update on subsequent runs).

### Studio is the single editing surface; secrets never enter Studio state

All terraform inputs (project id, region, GitHub repo, service name prefix,
runtime SA name override, ingress, container port, min/max instances,
gateway auth mode) are backed by Studio fields. Secret values flow through
the `setSecretValue` host action which shells `gcloud secrets versions add`
with the value piped through stdin; the value never enters Studio state,
React props, console logs, or argv.

### Stable Secret Manager naming

Secret Manager secret names are `${serviceNamePrefix}-${secretKey}` after
slugification, derived deterministically from the deployment plan by
`resolveSecretManagerName(plan, secretKey)`. The env var name (e.g.
`SUGARMAGIC_ANTHROPIC_API_KEY`) is independent of the secret name (e.g.
`wordlark-v1-abcde-anthropic-api-key`), bound through a `mappingHint` in
the plan so renames are tractable via terraform `moved {}` blocks. Single
shared secret set per game-version (no per-service-unit scoping in v1).

### One GCP project per game per major version per environment

Each game project carries a `majorVersion: number` field on `GameProject`.
The default `gcp_project_id` and `service_name_prefix` are templated as
`${gameSlug}-v${majorVersion}-${suffix}`. The suffix is a 5-character
lowercase base36 string generated client-side via `crypto.getRandomValues`,
recorded in the SugarDeploy plugin state slot, and preserved forever. The
collision-resistance is for GCP's globally-unique project id constraint;
the preservation is so worktrees and `git checkout v1.0.0` resolve back to
the original GCP project.

### SugarDeploy state lives in `pluginConfigurations[].config`

The deploy plugin's per-project state (deployment settings + the
per-major-version suffix map) lives in the SugarDeploy plugin's namespaced
`pluginConfigurations` config slot, NOT on `GameProject` directly. Reads go
through `getDeploymentSettings(gameProject)` and
`getVersionedProjectIdentifiers(gameProject)`; writes dispatch the generic
`UpdatePluginConfigurationCommand` produced by
`buildUpdateDeploymentSettingsCommand` /
`buildSetVersionedProjectIdentifierCommand`. The contractor test — "delete
`packages/plugins/src/catalog/sugardeploy/` and
`packages/plugins/src/deployment/`; does the domain package still
typecheck?" — passes because no domain code references SugarDeploy by
name except a single migration literal.

### Template-version drift handling is uniform

Generated terraform carries a `# SUGARMAGIC TEMPLATE VERSION: NN` stamp.
When the on-disk version is less than the current, the plugin emits the
full historical `moved {}` chain walked through the `TERRAFORM_RENAME_LEDGER`
so `terraform plan` stays non-destructive across renames. The Studio shows
a non-blocking drift banner until the next save regenerates the files at
the current version. There is no separate "N-2 or older" branch; N-1 and
N-5 take the same path. The ledger retains entries indefinitely for v1.

### Cut New Major Version is a Studio-orchestrated saga

Cutting a new major version is one user-visible operation composed of four
small host actions plus two in-memory domain dispatches, sequenced by the
Studio plugin workspace:

1. `prepare-cut-major-version` (host) — read-only pre-flight: git on PATH,
   work tree clean, `v{priorMajor}.0.0` tag doesn't exist.
2. `tag-prior-major` (host) — re-verifies pre-flight, then
   `git tag v{priorMajor}.0.0 HEAD`. First side effect.
3. `BumpMajorVersion` (domain command, dispatched by Studio) — sets
   `gameProject.majorVersion`. Plus a `buildSetVersionedProjectIdentifierCommand`
   dispatch that registers the new suffix in the plugin-state slot via
   `UpdatePluginConfiguration`.
4. `saveProjectWithManagedFiles` (Studio's existing IO path) — persists
   the bumped `project.sgrmagic` and regenerates the deploy.sh + terraform
   so the commit captures the full bump.
5. `commit-major-version-bump` (host) — `git add -u` + `git commit`.

On failure mid-saga the orchestrator runs best-effort rollback (untag for
persist-fail) and surfaces explicit manual recovery instructions when the
rollback itself can't complete (commit-fail). Push is never automatic.

### Local terraform state, gitignored

State lives at
`<game-root>/deployment/google-cloud-run/terraform/terraform.tfstate`
alongside `.terraform/`. Both are gitignored by the plugin-emitted
`.gitignore`. No GCS backend bootstrap problem for solo dev. A
remote-backend opt-in is a future epic, not part of this one.

### Same gateway scaffold across local and Cloud Run targets

`buildGatewayScaffoldFiles(targetId, unit, port, gatewayAuthMode)` produces
identical Dockerfile / server / package / routes output for both
`local` and `google-cloud-run`. The Cloud Run service runs the same
`node server.mjs` proxy the local Docker container runs. No bundled local
LLM, no bundled embeddings, no second runtime path.

### Two canonical cleanup paths

`gcloud projects delete <id>` is the nuclear path (entire GCP project
disappears, no terraform interaction). `terraform destroy` is the
fine-grained path (Artifact Registry / IAM / WIF / Secret Manager
containers removed, GCP project itself stays). The plugin's Teardown
Infra action runs `gcloud run services delete --quiet` first (because the
service is gcloud-owned and terraform doesn't know about it) and THEN
`terraform destroy` on the surrounding infrastructure.

## Consequences

- The Cloud Run service surface is fast-changing and lives outside
  terraform's plan/apply loop; surrounding infrastructure is slow-changing
  and lives inside. This split avoids the wordlark-v1 pattern where a
  placeholder service in terraform got overridden by `gcloud run deploy`
  on every CI run.
- Renaming a generated terraform resource means: bump
  `CLOUD_RUN_TEMPLATE_VERSION`, add an entry to `TERRAFORM_RENAME_LEDGER`
  describing the rename, ship. Old games regenerate with the full chain
  of `moved {}` blocks and `terraform plan` stays non-destructive.
- The deploy plugin can be uninstalled cleanly. Removing it from the
  workspace leaves the domain package typechecking clean. The
  game-authoring core (entities, sugarlang, dialogue, etc.) does not
  reference SugarDeploy types or commands.
- Major-version cuts are git-anchored. A `git worktree add` against an
  old tag re-opens that worktree's `project.sgrmagic` in Studio and
  SugarDeploy resolves back to the original GCP project for that major
  because the per-major suffix lives in the worktree's plugin-state slot,
  never deleted.
- Push to remote git or remote terraform state is never automatic. Cut
  the major locally, eyeball the commit, push when you mean to.
- The bearer-token gateway auth (`gatewayAuthMode: "bearer"`) is honor-
  system. The token is shared across all callers and lives in the game
  binary the user receives. It stops drive-by scrapers; it does not stop
  determined per-user abuse. Real per-user identity is [Plan 047](/docs/plans/047-sugarprofile-user-management-plugin-epic.md).
- Frontend / static-asset hosting is explicitly out of scope. This epic
  ships the gateway backend only. Making a deployed game playable in a
  browser requires a separate plan (player bundle + static hosting +
  gateway URL injection).

## Verification

- `packages/testing/src/plugin-infrastructure.test.ts` asserts the
  plugin's contributed host middleware list, the deployment target
  registry, the version-namespaced default project id derivation, the
  full lifecycle (deploy.sh + terraform tfvars shape), the secret
  manager naming rule, the cut-major-version domain dispatches, and the
  legacy-shape -> plugin-config-slot migration.
- Deleting `packages/plugins/src/catalog/sugardeploy/` and
  `packages/plugins/src/deployment/` should leave the domain package
  typechecking clean — the contractor test.
- A fresh wordlark save with `majorVersion: 1` and no overrides
  regenerates `deploy.sh` and `service.yaml` with `wordlark-v1-{suffix}`
  resource names.
- Hand-edit a `# SUGARMAGIC TEMPLATE VERSION: NN` stamp to a lower
  number; re-open in Studio. The drift banner appears. Save. Stamp
  matches current; banner clears.
- Cut wordlark from `v1` -> `v2` with a clean tree. After success:
  `git tag -l` includes `v1.0.0`, `git log -1` is the
  `chore: bump major version to 2` commit, the Version panel shows
  `v2 (active)` and `v1` with the original suffix preserved, and the
  resolved default project id is `wordlark-v2-{newSuffix}`.
