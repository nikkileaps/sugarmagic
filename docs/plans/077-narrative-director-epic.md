## Ground truth (verified 2026-07-22 against producing lines)

Code, not memory:

- **Quest domain** (`packages/domain/src/quest-definition/index.ts`): `QuestDefinition -> QuestStageDefinition -> QuestNodeDefinition` (behaviors objective/narrative/condition/branch). Nodes carry `onEnterActions` / `onCompleteActions: QuestActionDefinition[]`. `QuestActionType` union (L28-45) = `setFlag | giveItem | removeItem | playSound | spawnVfx | teleportNpc | moveNpc | setNpcState | emitEvent | unlockScene | advanceToNextScene | custom`. There is NO "player learns X" and NO "set time" action -- both are net-new (D and, for beat lifecycle, possibly G). `QuestStageState = "active" | "completed"`.
- **QuestManager** (`packages/runtime-core/src/quest/QuestManager.ts`): `getTrackedQuest()` returns `{ questDefinitionId, displayName, stageId, stageDisplayName, objectives[] }`. Observer hooks exist and are the seam for quest-gated events: `setEventHandler` (quest-start / stage-advance / quest-complete / objective-complete), `setStateChangeHandler`, `setActionHandler` (unhandled quest actions delegate here), `setNarrativeHandler` (L145-159).
- **Quest state reaches a conversation** via `runtimeBlackboardConversationMiddleware` (`coordination/gameplay-session.ts`, CONTEXT stage): it reads blackboard quest facts (synced every frame from `questManager.getTrackedQuest()`) and writes `runtimeContext.{ trackedQuest, activeQuestStage, activeQuestObjectives }`. **Sugarlang READS `activeQuestObjectives`** (`sugar-lang-context-middleware.ts:357-361` filters `questEssentialLemmas` by active objective `nodeId`). **Sugaragent does NOT** -- it reads only `trackedQuest.displayName` + `activeQuestStage.stageDisplayName`.
- **The firewall violation ships TODAY** (`.../generate/prompt/builder.ts:119-120`): every agent NPC's prompt gets `"The player is currently on a quest: \"{name}\" (stage: {stage}). This is the PLAYER's goal, not yours. Only reference it if the player brings it up..."`. So Finnick is already handed the quest title. Removing/replacing this is a concrete deliverable.
- **The teacher-director precedent (the pattern G mirrors)**: sugarlang's teacher middleware is a POLICY-stage `conversation.middleware` (`sugar-lang-teacher-middleware.ts:127`, `stage:"policy"`, priority 30). It computes a directive, builds an opaque `generatorPromptOverlay` string, and writes it to `execution.annotations["sugarlang.constraint"]`. `GenerateStage` reads that (`GenerateStage.ts:195`), passes `generatorPromptOverlay` as `promptContext.languageLearningOverlay`, and `builder.ts:203` splices it into the USER message -- the uncached per-turn half (Plan 072.4). Cost pattern (`sugar-lang-teacher.ts:60-108`): **cache-per-conversation -> try LLM -> deterministic fallback**; the fallback policy is pure lookup tables, zero LLM.
- **Middleware contribution is clean for a second middleware**: sugaragent already contributes a provider + the 073 memory middleware (`catalog/sugaragent/index.ts`). Stage order is `context -> policy -> generic -> analysis` (`runtime-core/conversation/index.ts:198`), then priority within a stage.
- **Quest-gated presence already exists**: `region-conditions/index.ts` `evaluateRegionQuestBinding({ questDefinitionId, stageId, worldFlagEquals })` -- consumed by behavior-task activation (`behavior/system.ts`) and collision/containment volumes. STAGE-level granularity. A conversation can drive world state today via the live action proposals `set-conversation-flag` (-> `questManager.setFlag`) and `notify-quest-event` (-> `questManager.notifyEvent`), and a dialogue node can fire `onEnterEventId` (`domain/dialogue-definition/index.ts:78`).

## The driving scenario -- "Find the Luggage"

The system must make THIS possible (nikki's canonical first quest):

1. Mim arrives at the station. Quest "Find the Luggage" begins.
2. Mim goes to the cargo dock to collect her suitcase. It is not there. A SCRIPTED NPC tells her to check baggage claim. From here she can go straight to baggage claim (next checkpoint) OR wander.
3. If she wanders to **Finnick** (an agentified cheese-shop NPC), it would be great if he -- WITHOUT knowing a priori that Mim lost a suitcase -- nudged her toward baggage claim ("I've seen bags routed down that way"), in his own gruff-but-warm voice, ONCE, and again if she returns. Every NPC she meets does the same in character. If it makes no sense for a character to engage, they just... talk, in character.
4. Meanwhile, quest-gated background events "happen": the suitcase has become sentient and wanders looking for socks to eat; a passenger at the cargo dock, upset about missing socks, appears -- but ONLY after arrival AND after Mim has talked to the scripted NPC.
5. She reaches baggage claim -> the stage advances -> the nudges retire; if the episode's quest ends and the next episode isn't out yet, every NPC is simply themselves again.

Two firewall constraints fall out of step 3-5: an NPC must never be told the player's objective verbatim, and "no quest engagement" must be a normal, common outcome -- not an error.

## Non-goals

- **A second LLM "drama manager."** Beat selection is a rules problem (see D1); v1 spends ZERO extra LLM calls. An LLM/async selection strategy is a deferred seam.
- **Emergent / simulationist social narrative** (the Versu pole: autonomous agents forming unauthored relationships). We author beats; we do not simulate drama. Powerful but uncontrollable, and wrong for a language game that must control what surfaces.
- **Strong mediation / railroading** (Mimesis-style block-or-replan). Cozy game -> weak mediation only: nudge and incorporate, never prevent a player action.
- **Handing NPCs the raw quest objective** (the firewall, D2).
- **A new quest-progression or presence engine.** Quest-gated world events COMPOSE existing quest actions + region-conditions + presence (D6).
- **Moderation / the judge / secrets-at-the-model-layer** (epic E/075). **Barks / audio** (epic F/076). **In-game-time authoring** (epic D/074).

## Design decisions (epic-review ratifies)

- **D1 -- Authored-first, rule-selected director; zero extra LLM calls (Strategy + Rules-Database patterns).** A NARRATIVE BEAT is the smallest unit of authored narrative direction (Facade's "beat" -- Mateas & Stern, AIIDE 2005). The director SELECTS at most one beat per turn by a deterministic rule: filter beats by precondition (quest active + stage + world flags + NPC-selector match + lifecycle) -> most-specific-wins -> else NONE. This is Elan Ruskin's shipped rule-database dialogue (L4D2/Dota: fuzzy-match world facts against authored rules, most-specific rule wins, else fall back / say nothing) and Facade's precondition-filter-then-select. It is a `Strategy` (the selection policy is swappable; v1 is the pure-rules policy, exactly like sugarlang's teacher FALLBACK policy which is lookup-only). The chosen beat's authored instruction is voiced by the SINGLE existing generate call -- no director LLM call. Rationale: cost (one cached call per turn, D7), author control (D3), and the field's consensus that even expensive drama optimization is precomputed OFFLINE, never run live (RL-DODM, Nelson et al.).
- **D2 -- The firewall: the raw objective NEVER enters the NPC prompt (Mediator pattern).** The director is a Mediator between quest ground-truth and the NPC: quest state (title, objective text, stage) is READ by the director and translated into a scoped, in-character NUDGE -- it is never forwarded to the model. This mirrors Inworld's `Instruction` action ("follow the instruction while maintaining their own personality" -- paraphrased) vs `say_verbatim`; the beat is an Instruction. Empirical justification: a system-prompt "must not reveal this under any circumstance" still leaked under ~10% of prompt injections (ProvSec 2025, arXiv 2508.19288) -- so "act like you don't know the quest" is NOT a safe container; the data must not be in the prompt. Concretely: DELETE `builder.ts:119`'s raw-quest line; the only quest-derived text the model ever sees is the director's authored nudge.
- **D3 -- "No directive" is a first-class output (Null Object / silence-as-output).** Most turns, the director returns null and the NPC responds exactly as it does today -- persona + world + memory, no quest awareness. Null is returned when: no active quest; the quest ended and no new one started (the between-episodes edge case nikki named); no beat is authored for this NPC at this stage; or a precondition is unmet. Precedent: Left 4 Dead's AI Director has an enforced "Relax" phase of deliberate non-intervention (30-45s); Ruskin's rule-DB degrades to silence on no-match; Versu treats inaction as legitimate. "Only where it makes sense" is therefore AUTHORED, not inferred -- an NPC engages the quest only where a human decided a beat should exist. This is both the cost win and the immersion win.
- **D4 -- Sugaragent concern; strict plugin separation via the shared neutral substrate (Blackboard + annotation seam).** The director lives in SUGARAGENT (generic dynamic-NPC direction; works with no sugarlang). Sugarlang's teacher-director lives in sugarlang (works in authored mode with no sugaragent). BOTH read the SAME neutral quest facts from `runtimeContext` (populated by runtime-core's blackboard middleware -- neither plugin owns quest state). Each emits its OWN opaque prompt overlay; they compose in the one generate call; there is NO cross-plugin import in either direction (the annotation seam is a string key, enforced by `check-plugin-catalog-boundary`). AGENTS.md: one source of truth (quest state = runtime-core), one-way deps (plugins depend on runtime-core, not each other).
- **D5 -- Beats are a first-class authored content type (`NarrativeBeatDefinition`).** A beat references a trigger (`questDefinitionId` + `stageId`, optional `worldFlagEquals`), an NPC selector (a specific `npcDefinitionId`, or a selector by role/tag/region so "any dock worker" can share a beat), an in-character `instruction` (the author's intent, phrased as guidance the NPC obeys in voice -- never the player's objective), a `lifecycle` (once-per-stage / repeat-on-return / retire-on-stage-advance), and a `specificity`/priority for most-specific-wins. AGENTS.md "one type per behavior": quests define PROGRESSION; beats define NPC DIRECTION -- distinct types. (Alternative weighed at review: co-locate `narrativeBeats[]` on `QuestStageDefinition`. Trade-off: co-location keeps the quest as one-source-of-truth for its own direction but bloats the quest type and can't express region-only/flag-only beats. Default to the separate type unless review shows co-location is cleaner.) Beats are game content, authored in Studio, consumed GENERICALLY by the director -- the lore-page precedent (neutral content; sugaragent is a consumer).
- **D6 -- Quest-gated world events COMPOSE existing machinery (Observer + condition-action), not a new engine.** The sentient suitcase / upset passenger are PRESENCE + BEHAVIOR gated on quest state, which already works: a quest node's `onEnterActions`/`onCompleteActions` (`setFlag` / `emitEvent`), the QuestManager observer hooks, region-condition bindings (`questStageId` AND `worldFlagEquals`), and presence/behavior gates. "Appears only after arrival AND after the NPC-1 conversation" is exactly `questStageId=X AND worldFlag(talkedToDockWorker)=true`, where the conversation sets the flag (`set-conversation-flag` proposal, or a scripted dialogue node's `onEnterEventId`). This story CLOSES only the smallest gaps: a clean helper for an agent conversation to set a quest/world flag, and the Studio authoring UX for compound (stage AND flag) triggers. No new gating engine.
- **D7 -- Cost + caching are load-bearing, and free here.** The director adds no LLM call. The nudge is spliced into the UNCACHED user half (Plan 072.4), so it never busts the cached persona/system prefix (Anthropic cache reads are 0.1x; a per-turn change to the cached prefix would be 1.25x-2x every turn). The selected beat is memoized per turn (and may be memoized per conversation if selection ever gets expensive -- the teacher-cache mold). Net per-turn cost delta vs today: zero.

## Stories (EXECUTION ORDER)

### 077.1 Narrative beat domain + authoring

The `NarrativeBeatDefinition` domain type per D5 (trigger, NPC selector, in-character instruction, lifecycle, specificity) + io parse/serialize + a Studio authoring surface (a Narrative Beats list; decide in-story whether it hangs off the quest editor or a dedicated view). Ship a starter beat for the Find-the-Luggage quest (Finnick + any dock NPC -> "nudge toward baggage claim"). Exit: domain + io unit tests (parse round-trip, selector shapes, lifecycle enum); the beat authorable and persisted in a game project.

### 077.2 The narrative director middleware (selection + firewall)

A sugaragent POLICY-stage `conversation.middleware` (`sugaragent.narrative-director`), contributed beside the memory middleware. It reads `runtimeContext` quest state + the memory annotation + the game's beats (from the boot payload / a resolver, the lore-load mold), runs the deterministic selector (D1: filter -> most-specific-wins -> lifecycle -> one-or-none), and writes `execution.annotations["sugaragent.narrative"]` = a `NarrativeDirective` with an in-character `directorOverlay: string` OR null. The raw objective/title never enters the overlay (D2). Beat lifecycle state (delivered-this-stage / delivered-count) is held in `execution.state` (the memoize-once mold, memory-middleware precedent) so once/repeat semantics work across turns without persistence in v1. Exit: unit tests -- selection (match, most-specific-wins, no-match -> null), lifecycle (once vs repeat-on-return), the firewall (assert the overlay never contains the objective/title text), and zero LLM calls.

### 077.3 Director into the prompt + Plan integration; remove the firewall breach

`GenerateStage` reads `execution.annotations["sugaragent.narrative"]` -> `promptContext.narrativeDirective` -> `builder.ts` splices it into the USER half (parallel to `languageLearningOverlay`, uncached per 072.4). **DELETE the raw-quest line at `builder.ts:119`** (D2). PlanStage: a turn carrying a beat counts as grounding -- route to the LLM to voice the nudge, and do not collapse to generic-only (the same shape as 073.3's `memoryGrounds`); no beat + no other grounding -> the NPC responds normally. Exit: integration test (mock gateway) -- an NPC with a matching beat delivers the nudge in character while the raw objective/title is absent from the prompt; an NPC with no beat (and the quest-over / no-active-quest cases) responds normally; the `builder.ts:119` line is gone.

### 077.4 Quest-gated world events (compose existing + close small gaps)

Wire the Find-the-Luggage background events on existing machinery (D6): the scripted dock-NPC conversation sets a `talkedToDockWorker` flag; the upset passenger is a presence/behavior gated on `stage=find-suitcase AND worldFlag(talkedToDockWorker)`. Close the two smallest gaps: (a) a clean, documented path for an agent (not just scripted) conversation to set a quest/world flag from a beat outcome or turn; (b) Studio authoring for the compound stage-AND-flag region binding. Scope tightly -- no new gating engine, reuse `evaluateRegionQuestBinding` + presence. Exit: in preview, the upset passenger spawns ONLY after arrival AND after the dock-NPC conversation, and despawns/retires at stage advance; a unit test on the compound binding evaluation.

### 077.5 Config, dev inspection, cost guard, wrap

Config: a `narrativeDirectorEnabled` flag (off -> the middleware is a no-op, NPCs respond as pre-G). Dev handle `window.__sugaragentDirector`: dump the current turn's selection -- which beat was chosen and why, or which preconditions failed for every candidate (the house debug-tooling style, memory-debug mold). Cost guard: a test asserting the director adds ZERO gateway/LLM calls. Epic wrap: `docs/api` (beat authoring contract, the director annotation, the firewall principle, the compose-don't-reinvent note for world events); deferred-seam sweep. Exit: settings render via the schema auto-renderer; dev handle documented; cost test green.
