## Ground truth (verified 2026-07-22 against producing lines)

Code, not memory:

- **Quest domain** (`packages/domain/src/quest-definition/index.ts`): `QuestDefinition -> QuestStageDefinition -> QuestNodeDefinition` (behaviors objective/narrative/condition/branch). Nodes carry `displayName`, `description`, `targetId`, `dialogueDefinitionId`, `voiceoverText`, `eventName`, `onEnterActions`/`onCompleteActions: QuestActionDefinition[]`. `QuestActionType` (L28-45) = `setFlag | ... | emitEvent | unlockScene | advanceToNextScene | custom` (no "player learns" / "set time"). `QuestStageState = "active" | "completed"`.
- **QuestManager** (`packages/runtime-core/src/quest/QuestManager.ts`): `getTrackedQuest()` -> `{ questDefinitionId, displayName, stageId, stageDisplayName, objectives[] }`; objectives are `{ nodeId, displayName, description, showInHud, optional }`. Observer hooks for world events: `setEventHandler` / `setStateChangeHandler` / `setActionHandler` / `setNarrativeHandler` (L145-159). A conversation can drive world state via live action proposals `set-conversation-flag` (-> `setFlag`) and `notify-quest-event` (-> `notifyEvent`); a dialogue node fires `onEnterEventId` (`domain/dialogue-definition/index.ts:78`).
- **Quest state already reaches conversations**: `runtimeBlackboardConversationMiddleware` (`coordination/gameplay-session.ts`, CONTEXT stage) writes `runtimeContext.{ trackedQuest, activeQuestStage, activeQuestObjectives }` from blackboard facts synced each frame from `getTrackedQuest()`. Sugaragent's RetrieveStage folds `trackedQuest.displayName` into the search query ("Active quest: ...") ONLY on a turn the interpreter classified `intent === "quest_guidance"` (`RetrieveStage.ts:150`, gated by `hasQuestGuidanceCue(playerText)` in `interpretation.ts`) -- i.e. only when the player's text already cues quest talk. It does NOT fold `activeQuestStage.stageDisplayName`, and does NOT use the active OBJECTIVE (nodeId/displayName/description/targetId) at all. So on a turn where Mim never says "baggage", nothing quest-relevant is folded -- which is exactly the gap D3 closes. Sugarlang reads `activeQuestObjectives` for vocab (`sugar-lang-context-middleware.ts:357`).
- **The firewall violation ships TODAY** (`packages/plugins/src/catalog/sugaragent/runtime/stages/generate/prompt/builder.ts:119-120`, in `buildWorldStateUserLines`, the uncached USER half): every agent NPC's prompt gets `"The player is currently on a quest: \"{name}\" (stage: {stage}). This is the PLAYER's goal, not yours..."`. No test pins that sentence, but `builder.test.ts:103` asserts the raw quest displayName (`"Lost Locket"`) IS present in the user prompt -- 077.1 inverts that assertion. Replacing this is a concrete deliverable (D2).
- **The blackboard (Plan 023)** (`runtime-core/src/state/blackboard.ts`): `setFact` / `getFact` / `subscribe`; scopes `global | region | entity | quest | conversation`; lifecycles `persistent | session | frame | ephemeral`; plugins register fact definitions (sugarlang has `SUGARLANG_BLACKBOARD_FACT_DEFINITIONS`). This is the shared world-narrative-state substrate (D4).
- **The blackboard write firewall (decisive for D4)**: `setFact` calls `assertWriteAllowed` (`blackboard.ts:484`), which THROWS unless `definition.ownerSystem === options.sourceSystem`. Every existing quest/world fact is `ownerSystem: "quest-system"` (`blackboard.ts:253-273`), written ONLY by runtime-core's `syncBlackboardQuestFacts` (`gameplay-session.ts:835`). Sugarlang owns and writes only `sugarlang.*`-namespaced facts, and it does so from its own middleware/reducer which holds a `RuntimeBlackboard` handle (`sugar-lang-context-middleware.ts:276`). **Sugaragent's conversation stages have NO blackboard handle** -- they receive only `execution` (annotations/state). The ONLY channel from an agent turn to world state is `ConversationActionProposal` (`conversation/index.ts:82`), whose live kinds are `set-conversation-flag | notify-quest-event | start-scripted-followup | request-close`; PlanStage already emits proposals (`PlanStage.ts:56`), which flow via `DialogueManager` onTurn to `handleConversationActionProposal` (`gameplay-session.ts:1069`) for free-form conversations too. There is NO proposal kind that writes a blackboard fact today -- so D4's "NPCs WRITE it" needs a new runtime-core-owned write path (see 077.3a), not an existing seam.
- **Quest-gated presence already exists**: `region-conditions/index.ts` `evaluateRegionQuestBinding({ questDefinitionId, stageId, worldFlagEquals })`, consumed by behavior-task activation and collision/containment volumes. Stage-level. This is the seam for quest-gated world events (D5) -- no new engine needed.

## The driving scenario -- "Find the Luggage"

The system must make THIS possible:

1. Mim arrives; quest "Find the Luggage" begins.
2. She goes to the cargo dock; her suitcase isn't there; a SCRIPTED NPC tells her to check baggage claim. From here she can go straight there OR wander.
3. If she wanders to Finnick (agentified cheese-shop NPC), it'd be great if he -- WITHOUT knowing a priori that Mim lost a suitcase -- nudged her toward baggage claim in his own voice, if it fits, and again if she returns. Every NPC does the same in character; if it makes no sense for a character to engage, they just talk.
4. Background events "happen": the suitcase has gone sentient and wanders eating socks; a passenger at the cargo dock, upset about missing socks, appears -- but ONLY after arrival AND after Mim has talked to the scripted NPC.
5. She reaches baggage claim -> stage advances -> nudges retire. Episode's quest over with no next quest -> every NPC is simply themselves again.

## What this is (and the thing it is NOT)

There is **no Narrative Director component.** The "writer brain watching the board" is realized by three things that already exist, working together:

1. **The agent's own pipeline** gathers facts and responds (Interpret -> Retrieve -> Plan -> Generate). We make it QUEST-AWARE (D3): while a quest is active, the NPC's Retrieve pulls quest-relevant world lore and the objective's target is available -- so Finnick has "lost luggage -> baggage claim" even if Mim never says "baggage". Loaded once at conversation start, memoized (persona/memory mold), so it's not heavy per turn.
2. **The blackboard** holds shared WORLD-NARRATIVE STATE (D4): where the sentient suitcase is, how many times the baggage-claim goal has been surfaced, which flags have flipped. NPCs READ it (it rides in their prompt); the WRITE happens runtime-side (a new proposal kind whose handler does the owner-side `setFact`) driven by a turn's outcome -- sugaragent never touches the blackboard. Cross-NPC coordination is implicit through this shared state -- NPC A hints (prompted), the counter bumps, NPC B sees it and eases off (best-effort, since the proxy counts prompting not saying). No central mind; the field's "town-wide awareness" pattern is shared memory + per-agent reasoning (Generative Agents).
3. **The quest + world-event system** makes the WORLD change (D5): spawns the upset passenger, moves the suitcase, advances the stage. This is the only genuinely central "director," it is DETERMINISTIC and AUTHORED (quest actions + region-conditions + presence), and it already exists.

Per-NPC JUDGMENT ("should Finnick mention it now, in character?") is delegated to Finnick's own generate call, informed by the shared state in his prompt + his persona. That is both cheaper and more in-character than any central brain deciding for him.

## Non-goals / Rejected (on merit)

- **A central LLM narrative director is REJECTED, not deferred.** A brain dictating each NPC's quest line (a) fights the agent architecture (a foreign intelligence makes NPCs LESS in-character), (b) is a coherence RISK not a win (runtime LLM global-narrative decisions are where coherence dies -- AI Dungeon / drama-manager demos), and (c) costs an extra call for a worse result. Coherence comes from authored structure + shared state. We say no because it won't do the job right. (The ONLY "central director" that belongs is the quest/world-event system -- deterministic, authored -- which we already have and use here.)
- **Watchlist, not a plan: global pacing.** A Left-4-Dead-style pacing controller ("ease off, NPCs are piling on / escalate, the player's stuck") is the one place a director-like layer has real merit -- but it's rhythm, not dialogue, and a cozy author-paced episodic game paces via its quest/episode structure. The one distributed pacing worry (too many NPCs piling on) is a shared counter (D4), not a brain. Revisit only if Wordlark ever becomes a tense, systemic game it currently is not; it would be a thin layer ON TOP of the distributed NPCs.
- **Authored per-NPC beat/hint content is NOT in v1.** Controlled, must-happen narrative moments are already AUTHORED as scripted content (quest narrative nodes, scripted dialogue). The agentified layer is the SOFT, emergent one -- persona + facts + prompt. If emergence proves too loose in play, light authored gating is the deferred escape hatch (below), not the starting point.
- Emergent social simulation (the Versu pole). Moderation / secrets-at-the-model-layer (epic E/075). Barks/audio (epic F/076). In-game-time authoring (epic D/074).

## Design decisions (epic-review ratifies)

- **D1 -- No director; the "writer brain" is authored quests + shared blackboard state + per-NPC agents.** See "What this is" above. This is the architecturally correct decomposition and the one that scales (add NPCs/quests without a central bottleneck).
- **D2 -- The firewall is about SECRETS, not the quest goal (nudge != secret).** A secret (must never be revealed) stays out of the prompt entirely (empirically, a system-prompt "never reveal" leaks ~10% under injection -- ProvSec 2025). A quest NUDGE is the opposite: we WANT it (optionally, naturally) said, so it may be in the prompt -- what matters is FRAMING. So: DELETE `builder.ts:119-120`'s "the player's goal, not yours" line; replace with WORLD-framed context + instruction ("here is what would be helpful in the world right now; offer help you'd plausibly know if it fits your character; do NOT act as though you know the player's private business; do not repeat yourself"). The model never sees "the player is looking for their suitcase"; it sees "travelers with lost luggage are directed to baggage claim."
  - **THE PROMPT INVARIANT (adjudicates the D2/D3 tension -- load-bearing).** The quest objective's authored `displayName`/`description` (e.g. an objective literally named "Find your lost suitcase") ARE the player's private goal. They may be used INTERNALLY -- to seed a retrieval query or resolve which world-lore/target to surface -- because a query string is never shown to the model. They must NEVER be spliced into the built prompt verbatim. What enters the prompt is ONLY: (a) world-framed lore retrieved via that internal query ("travelers with lost luggage are directed to baggage claim"), and (b) the objective's `targetId` as a neutral world referent (an area/node id like `baggage-claim`), never its private-goal prose. This invariant is the seam between D2 (never the private objective) and D3 (feed the objective to reach the fact): D3 feeds the objective INTO retrieval; only its downstream world-lore + target referent come back OUT into the prompt.
- **D3 -- Quest-relevant facts made reachable while a quest is active (the real substance).** Today Retrieve folds the quest name only on a player-text-cued `quest_guidance` turn (see Ground truth), so if Mim never says "baggage" nothing surfaces. Fix, following the memory-middleware mold (`memory-middleware.ts` -- a CONTEXT-stage middleware that loads once, memoizes in `execution.state`, republishes a per-turn annotation): a sugaragent CONTEXT-stage middleware resolves the active objective's target world-lore/fact ONCE at conversation start (keyed by the objective per the D2 prompt invariant -- objective text seeds resolution but is not itself surfaced), memoizes it, and republishes it as an annotation that GenerateStage splices into the prompt as world-framed context. This is the seam that makes "load once, memoize, refresh only on quest-state change" coherent -- it does NOT try to bias the per-turn, player-text-driven Retrieve vector query (which is the wrong seam for a load-once fact). Pattern: reuse the memory/annotation middleware precedent; no new subsystem.
- **D4 -- Shared world-narrative state on the blackboard; NPCs READ it, runtime-core WRITES it on their behalf.** A `goal-surfaced-count` fact (how many times the current objective has been raised to the player) and any world-entity flavor facts (e.g. suitcase-location) live on the blackboard. They ride into the prompt (the NPC reads them). **Ownership + write path (per the blackboard write firewall in Ground truth):** these are `world/narrative` state, so runtime-core owns them (`ownerSystem: "quest-system"`, or a new `"narrative-system"` if a distinct owner is cleaner) -- NOT sugaragent, and NOT a `world.*` fact owned by a plugin (that would violate D6 one-source-of-truth). A sugaragent turn does NOT write the fact directly; it emits a NEW `ConversationActionProposal` kind (e.g. `note-world-narrative` / `bump-goal-surfaced`) whose HANDLER in `gameplay-session.ts` performs the owner-side `setFact` with the runtime-core `sourceSystem`. This reuses the existing proposal channel end-to-end and keeps sugaragent from ever touching the blackboard. Building this write path is 077.3a (its own story). HONEST WRINKLE: knowing "did the NPC actually say it?" is hard (LLM output is prose) -- v1 uses a COARSE proxy that counts PROMPTING, not saying (bump when PlanStage routed the turn to voice quest context, i.e. we asked the model to steer -- the model may still decline in character). So the count can bump on a turn where no nudge was actually delivered; "second NPC eases off" is therefore BEST-EFFORT emergent, not a guarantee. PRECISE "was the beat delivered, in character?" is deferred to epic E/075's judge. Pattern: Blackboard + the existing proposal channel.
- **D5 -- Quest-gated world events COMPOSE existing machinery (Observer + condition-action).** The sentient suitcase / upset passenger are presence + behavior gated on quest state: quest node `onEnter`/`onComplete` actions (`setFlag`/`emitEvent`), QuestManager observer hooks, region-condition bindings (`questStageId` AND `worldFlagEquals`), presence/behavior gates. "After arrival AND after the NPC-1 conversation" = `stage=find-suitcase AND worldFlag(talkedToDockWorker)`, the flag set by that conversation (`set-conversation-flag` / a scripted dialogue node). Close only the smallest gaps (a clean flag-set path from an agent turn; Studio authoring for compound stage-AND-flag triggers). No new gating engine.
- **D6 -- Plugin separation holds (Blackboard + annotation seam).** All the above is SUGARAGENT + runtime-core; it works with no sugarlang. Sugarlang independently reads the SAME neutral quest facts for vocab. No cross-plugin import (catalog-linter enforced -- but note `check-plugin-catalog-boundary.mjs` only blocks cross-catalog RELATIVE imports; it does NOT police fact ownership or `world.*` naming, so the D4 ownership discipline (world-narrative facts owned by runtime-core, written via a proposal handler, per 077.3a) is a manual design invariant, not linter-caught). One source of truth: quest/world state = runtime-core.
- **D7 -- Cost + caching, free here.** Zero extra LLM calls: the quest context + shared state are voiced by the single existing generate call, spliced into the UNCACHED user half (Plan 072.4) so the cached persona/system prefix is never busted (cache reads 0.1x). Quest-relevant facts load ONCE per conversation, memoized. Per-turn cost delta vs today: zero.

## Stories (EXECUTION ORDER)

### 077.1 Quest context into agentified dialogue (frame it right; delete the breach)

Replace `builder.ts:119-120`'s omniscient quest line with WORLD-framed quest context + an instruction block per D2, in the uncached user half: the objective phrased as "what would be helpful in the world right now" (NOT its raw title -- see the D2 prompt invariant), plus "offer help you'd plausibly know if it fits your character; don't act as though you know the player's private business; don't repeat what's already been said." PlanStage: a turn where quest context is relevant is grounded (routes to the LLM to voice it), like 073.3's `memoryGrounds`; otherwise the NPC responds normally. Note: `builder.test.ts:103` currently ASSERTS the raw quest displayName ("Lost Locket") is present in the user prompt -- that assertion must FLIP to assert its ABSENCE. Exit: integration test (mock gateway) -- an NPC offers the baggage-claim steer in character while the raw objective/title is ABSENT from the prompt (positive regression guard on the firewall, which is otherwise untested after the delete); an NPC with no active quest (and the quest-over case) responds normally; `builder.ts:119-120` is gone; `builder.test.ts:103` flipped.

### 077.2 Quest-relevant facts reachable while a quest is active (the substance)

Add a sugaragent CONTEXT-stage middleware (the memory-middleware mold, D3): while a quest is active, resolve the active objective's target world-lore/fact ONCE at conversation start (the objective seeds resolution per the D2 prompt invariant; its displayName/description are NEVER surfaced), memoize it in `execution.state`, and republish an annotation GenerateStage splices into the UNCACHED user half (D7) as world-framed context. Memo key + invalidation (unlike memory's present-or-absent memo, this one must refresh when the quest changes): key the memo on the active `trackedQuest.questId` + `activeQuestStage.stageId` from `execution.runtimeContext` (available each turn per `conversation/index.ts`), and re-resolve when either changes. Do NOT bias the per-turn Retrieve vector query (wrong seam for a load-once fact). So Finnick has "lost luggage -> baggage claim" even if Mim never says "baggage". Note: the once-per-quest-state resolution may perform a lore/vector LOOKUP (network I/O, NOT an LLM completion) -- off the per-turn path and outside 077.5's "zero extra LLM calls" guard, which is about completions. Exit: integration test -- with the Find-the-Luggage quest active and a lore page about baggage claim, an NPC's prompt contains the baggage-claim world fact WITHOUT the player mentioning it, AND the raw objective displayName/description is ABSENT from the prompt (the prompt invariant); with no quest active, neither appears. Assert the fact is resolved once per quest-state (memoized), not per turn, and re-resolved when the stage advances.

### 077.3a Runtime-side write path for world-narrative facts (the missing mechanism)

The blackboard write firewall (Ground truth) means an agent turn CANNOT write a blackboard fact today: sugaragent stages hold no blackboard handle, `assertWriteAllowed` requires `ownerSystem === sourceSystem`, and no `ConversationActionProposal` kind writes a fact. Build the path: (1) define the world-narrative facts owned by runtime-core (`goal-surfaced-count`, any world-entity flavor facts) -- `ownerSystem` is a runtime-core system (`quest-system` or a new `narrative-system`), NOT sugaragent; (2) add a new `ConversationActionProposal` kind (e.g. `note-world-narrative` / `bump-goal-surfaced`) to the union in `conversation/index.ts` and a handler branch in `handleConversationActionProposal` (`gameplay-session.ts`) that performs the owner-side `setFact` with the runtime-core `sourceSystem`; (3) sugaragent's PlanStage emits that proposal when it routed the turn to voice quest context (the coarse proxy). Sugaragent never touches the blackboard. Exit: unit test -- a `note-world-narrative` proposal handled by runtime-core writes the fact with the owner `sourceSystem` (and a direct sugaragent `setFact` would throw, documented as why the proposal indirection exists); the proposal-union `never`-exhaustiveness still compiles.

### 077.3 Shared world-narrative state (blackboard) NPCs read; eased-off coordination

Depends on 077.3a (the write path). Wire the read side + the coarse proxy: NPCs read the world-narrative facts (into the prompt, world-framed), and PlanStage emits the write proposal (077.3a) when it prompted the model to voice quest context. This gives "mention once, again on return, and other NPCs ease off" without a central brain -- BEST-EFFORT, since the proxy counts PROMPTING not saying (D4): the count can bump on a turn where the model declined to steer. Exit: unit + integration -- after a turn where PlanStage routed to voice quest context, the count bumps (via the 077.3a path); a second NPC's prompt reflects the raised count; no persistence beyond session in v1. Do NOT assert "the NPC said it" (unknowable in v1; deferred to E/075).

### 077.4 Quest-gated world events (compose existing + close small gaps)

Wire the Find-the-Luggage background events on existing machinery (D5): the scripted dock-NPC conversation sets `talkedToDockWorker` via the EXISTING `set-conversation-flag` proposal (`gameplay-session.ts:1073` -> `questManager.setFlag`); the upset passenger is presence/behavior gated on `stage=find-suitcase AND worldFlag(talkedToDockWorker)`. `evaluateRegionQuestBinding` already evaluates `questStageId AND worldFlagEquals` as a conjunction (`region-conditions/index.ts:84-110`) -- no engine change. Close the two smallest gaps: (a) DOCUMENT the existing agent-turn flag-set path (this is a quest FLAG via `set-conversation-flag`, distinct from 077.3a's world-narrative FACT write -- do not conflate them); (b) the Studio authoring for a compound stage-AND-flag binding. FIRST verify whether the authoring UI already exposes both clauses -- if it does, (b) is documentation only; if it does NOT, split "add compound-binding authoring" into its own tracked sub-task (per story-scope-completeness -- do not let a secretly-two-story exit ship under one label). No new gating engine. Exit: an integration test on `evaluateRegionQuestBinding` composed with presence activation -- the passenger's presence resolves ONLY when `stage=find-suitcase AND worldFlag(talkedToDockWorker)` both hold, and retires when the stage advances (automated, not preview-only); plus a preview smoke pass of the full arrive -> dock-convo -> passenger-appears -> stage-advance -> retire loop.
