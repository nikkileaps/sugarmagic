# Plan 045: SugarDeploy Cloud Run Plugin-Owned Infrastructure Epic

**Status:** Accepted
**Date:** 2026-05-18 (proposed) / 2026-06-15 (accepted)

## Epic

### Title

Complete the SugarDeploy Cloud Run target so that every artifact required to stand up, deploy, observe, rotate, and tear down a game's cloud infrastructure is generated and orchestrated by the plugin. No hand-written terraform, no per-game bespoke deploy scripts, no third repo, no CLI-only secret-setting workflow. The Studio SugarDeploy workspace becomes the single editing and operating surface for the full deployment lifecycle on Google Cloud Run.

### Goal

- **Plugin-generated terraform under the deployment directory.** The SugarDeploy plugin emits `main.tf`, `variables.tf`, `outputs.tf`, and `terraform.tfvars` into `<game-root>/deployment/google-cloud-run/terraform/` on every project save. Every file carries the existing `GENERATED BY SUGARMAGIC - DO NOT EDIT` header AND a new `# SUGARMAGIC TEMPLATE VERSION: NN` stamp. Terraform owns the **slow-changing surrounding infrastructure**: Artifact Registry repo, runtime service account, Workload Identity Federation pool/provider scoped to the configured GitHub repo, IAM bindings (run.admin, artifactregistry.writer, secretmanager.viewer for the GitHub WIF principal; secretAccessor for the runtime SA), and empty Secret Manager containers for vendor API keys. Terraform deliberately does **NOT** own the `google_cloud_run_v2_service` resource — the Cloud Run service lifecycle is `gcloud`-managed (see deploy and stop below). This avoids the wordlark-v1 awkwardness of declaring a placeholder service in terraform that CI immediately overrides via `gcloud run deploy`, and keeps fast-changing deploy state out of terraform's plan/apply loop.

- **Studio fields back every terraform input.** The SugarDeploy workspace's Cloud Run section grows fields for `gcp_project_id`, `region`, `github_repo`, `service_name_prefix`, and any other terraform variables. Values persist in `gameProject.deployment.targetOverrides["google-cloud-run"]` (same path the existing overrides already use). No `.tfvars` hand-editing is part of the supported workflow.

- **Local terraform state, gitignored.** State lives at `<game-root>/deployment/google-cloud-run/terraform/terraform.tfstate` alongside `.terraform/`. Both are gitignored. No GCS backend bootstrap problem for solo dev. A remote-backend opt-in is a follow-up epic, not part of this one.

- **Same gateway scaffolding shared across local and Cloud Run targets.** Verify and document that `buildGatewayScaffoldFiles(targetId, unit, port)` produces identical Dockerfile / server / package / routes output for `local` and `google-cloud-run`. Eliminate any subtle divergence. The Cloud Run service runs the same `node server.mjs` proxy the local Docker container runs — no bundled local LLM, no bundled embeddings, no second runtime path.

- **Cloud Run deploy script consumes terraform outputs.** The generated `deploy.sh` reads Artifact Registry URL, runtime SA email, and Secret Manager secret names from `terraform output -json`. The script itself owns the Cloud Run service lifecycle via `gcloud run deploy` (idempotent — creates on first run, updates on subsequent runs). Secrets attach via `--set-secrets=KEY=SECRET_NAME:latest`, not `--set-env-vars`. Non-secret config (model names, vector store id, etc.) attach via `--set-env-vars`.

- **Studio-driven secret value management.** The SugarDeploy workspace's Cloud Run section grows a Secrets subsection listing every secret declared in the deployment plan. Each secret row shows a status (Set / Not set, last-updated timestamp + version id) and a Set Value action. Set Value opens a password-masked modal; submit sends a new host action `setSecretValue` to the Studio dev server middleware, which shells `gcloud secrets versions add` with the value passed via stdin. Values never enter studio state, never log, never persist beyond the gcloud invocation.

- **Stable Secret Manager naming convention.** Secret Manager secret names follow `${serviceNamePrefix}-${secretKey}` after slugifying, derived deterministically from the deployment plan. Env var name (e.g. `SUGARMAGIC_ANTHROPIC_API_KEY`) stays independent of the secret name (e.g. `wordlark-gateway-anthropic-api-key`) — bound through a `mappingHint` declared in the plan so renames become tractable via terraform `moved {}` blocks. Per-plan scoping (one set of secret values shared across all service units in a plan) is the v1 default; per-unit scoping is a future extension that prepends the service unit id when needed.

- **Lifecycle host actions for the full loop.** Extend the existing `/__sugardeploy/action` middleware (`apps/studio/vite.config.ts`) with action kinds: `setup-infra` (terraform apply), `teardown-infra` (delete Cloud Run service first via `gcloud run services delete --quiet`, THEN `terraform destroy` — ordering is required because terraform doesn't know about the service but its IAM/WIF bindings are referenced by it), `set-secret-value`, `secret-status`, plus the existing `deploy` / `status`. Implement `stop` for Cloud Run as `gcloud run services delete <service-name> --quiet` (mirrors `docker compose down` for the local target — service gone, surrounding infra and image remain so a subsequent `deploy` is fast). Implement `health` for Cloud Run as a real HTTP probe (not just a returned URL) against the deployed service URL retrieved via `gcloud run services describe`; upgrade the local target's `health` to perform an actual probe too, so both targets behave symmetrically. Each action is bound to a Studio UI affordance in the SugarDeploy workspace.

- **Template-version drift handling.** Plugin records the current template version it emits. On regeneration, if the existing artifacts on disk carry an older version, the plugin emits terraform `moved {}` blocks for any resource renames between versions so `terraform plan` stays non-destructive for existing games.

- **One GCP project per game per environment, documented as canonical.** The plugin-generated README and the SugarDeploy workspace copy state explicitly that cross-game resource sharing inside one GCP project is not supported. `gcloud projects delete` is the canonical nuclear cleanup; `terraform destroy` is the canonical fine-grained cleanup.

- **Game-project versioning via `majorVersion` metadata + git tags + version-namespaced GCP projects.** Each game project carries a `majorVersion: number` field in its top-level metadata. The deployment plugin's default `gcp_project_id` and `service_name_prefix` are templated as `${gameSlug}-v${majorVersion}` (override field still wins for the wordlark-style "reuse the existing GCP project" case). Cutting a new major version is a first-class Studio affordance: a **Cut New Major Version** button performs a sequenced `git tag` + metadata bump + `git commit` so the old major is anchored in git history and subsequent deploys land in a fresh GCP project. The old version's GCP project keeps running untouched because nothing redeployed it. Patching an old major is `git worktree add` against the old tag → open that worktree in Studio → its metadata says the old `majorVersion` → SugarDeploy targets the old GCP project automatically. One directory, one repo, one source of truth (git), N coexisting deployed versions.

### Why this epic exists

Earlier game projects shipped hand-written terraform alongside the game code. The engine evolved; the per-game terraform did not. Drift between engine expectations and game-root infrastructure produced parity bugs and stale cloud resources that this team has lived through (the wordlark-v1 to wordlark migration is the immediate trigger). Plan 021 and ADR 009 established the principle that Sugarmagic owns generated deployment artifacts even when they live in the game root. The local target already proves this works (plugin-generated Dockerfile, `server.mjs`, `docker-compose.yml`, all overwritten on save, no drift possible). This epic extends the same property to Cloud Run infrastructure — the highest-friction drift surface in the old setup.

The existing `google-cloud-run` target generates the gateway scaffold, a `deploy.sh`, a `service.yaml`, and a README, but assumes the GCP-side infrastructure (Artifact Registry, IAM, WIF, Secret Manager) already exists. The user fills in those gaps by hand today; that hand-work IS the drift surface. Closing it makes Cloud Run targets reproducible and disposable per game.

The decision to keep the same Dockerfile and gateway across local and Cloud Run (no bundled local LLM/embeddings in the cloud image) preserves AGENTS.md's "single enforcer / one type per behavior" rule. Bundling models for cloud and proxying for local would be exactly the kind of split-brain architecture this project exists to prevent.

The decision to drive secret values from the Studio (not CLI-only) follows the same single-input-surface logic that already governs Working Directory, project ID, and region: one place to edit, one validation path, one mental model.

The decision to support fine-grained (`terraform destroy`) and nuclear (`gcloud projects delete`) cleanup paths together — under a one-project-per-game/env discipline — directly addresses the existing stale-infra problem the team is currently living with.

### What is NOT in scope

- AWS Fargate or other non-Cloud-Run hosted targets (separate future epic per target)
- Multi-environment (staging vs production) per game — single environment per game project for v1; multi-env is a follow-up epic that may layer on `deployment.environmentOverrides` or move to a per-env subdirectory layout
- GitHub Actions workflow generation — the plugin's `deploy` action invoked from the Studio is the supported deploy path for v1; CI generation can land as a follow-up
- Frontend / Netlify deploy — out of scope; the Cloud Run target covers the backend gateway only
- Remote terraform state backend (GCS) — local state only for v1; remote-backend opt-in is a separate follow-up
- Custom domain / DNS / load balancer setup — Cloud Run-provided `*.run.app` URL only for v1
- A migration tool to import existing wordlark-v1-style infrastructure into the new plugin-generated terraform — wordlark-v1 stays as reference; new games start clean

- extend `normalizeGoogleCloudRunDeploymentTargetOverrides` (and the underlying type in `packages/domain/src/deployment/`) with new fields: `githubRepo` (persisted exactly as `owner/repo` — matches the GitHub OIDC `repository` claim so terraform IAM principals and `attribute.repository == "owner/repo"` conditions work with no transformation), optional `runtimeServiceAccountName`, anything else terraform needs
- add `majorVersion: number` to the top-level game project metadata type in `packages/domain/src/project/` (default to `1` when reading projects that lack the field — zero-effort migration for existing wordlark and any other in-flight projects). The override field continues to win when present (escape hatch for "reuse an existing arbitrarily-named GCP project" like the legacy `wordlark` project). Both `resolveSecretManagerName` and the deploy-script generator consume the resolved (overridden-or-defaulted) `serviceNamePrefix` — they do not derive version from elsewhere
- add a `cutNewMajorVersion(gameRoot)` helper in `packages/plugins/src/deployment/` (or a more general project-lifecycle module if one already fits) that performs the sequenced operation: (1) pre-flight checks — `git` on PATH, working tree clean, the tag `v${currentMajor}.0.0` does not already exist; (2) `git tag v${currentMajor}.0.0` at current `HEAD` (anchors the last commit of the old major); (3) bump `majorVersion` to `currentMajor + 1` in the project metadata file and write it; (4) `git add <metadata-file>` + `git commit -m "chore: bump major version to ${newMajor}"`. On any failure mid-sequence, roll back: delete the tag created in step 2 (`git tag -d`), `git checkout -- <metadata-file>` to discard the write from step 3. Push is **not** automatic — solo dev pushes their own tag/commit when ready
- add template version stamping helper + `moved {}` emission discipline; document the convention in `packages/plugins/src/deployment/README.md`
- add a `TERRAFORM_RENAME_LEDGER` constant in `packages/plugins/src/deployment/` as the source of truth for cross-version resource renames: `Array<{ fromVersion: number, toVersion: number, moves: Array<{ from: string, to: string }> }>`. On every regeneration the plugin walks the ledger from the on-disk version up to the current version and emits the union of `moved {}` blocks needed into `main.tf`. For v1 of this epic the ledger is empty (no prior versions exist); all entries are added in future epics when renames actually happen. The ledger retains entries indefinitely for v1 — a future plan may cap retention once the version count makes the generated terraform unwieldy
- expose a small read helper `readTemplateVersionFromDisk(gameRoot)` that parses the `# SUGARMAGIC TEMPLATE VERSION: NN` stamp from the existing `main.tf` (if present); returns `null` when no terraform has been generated yet. The Studio uses this to drive the drift banner described in the Studio UI section
- add a shared helper `resolveSecretManagerName(plan, secretKey)` in `packages/plugins/src/deployment/` that produces `${serviceNamePrefix}-${secretKey}` with these rules: lowercase the input, replace any char NOT in `[a-z0-9-]` with `-`, collapse consecutive `-`, trim leading/trailing `-`, and if the result starts with a digit, prepend `s-` (GCP Secret Manager names cannot start with a digit). If `serviceNamePrefix` is empty, fall back to the game project slug; if neither is resolvable, throw a plan-time error. Cap the final name at 80 characters and throw a clear error if exceeded (GCP allows 255 but CLI/Cloud Run references get unwieldy past ~80). This helper is the single source of truth used by the terraform generator, the deploy script generator (for `--set-secrets` flags), and the `set-secret-value` / `secret-status` host actions — those four call sites must not derive the name independently or they'll drift

### Host actions (Studio dev server middleware)

- extend `createSugarDeployHostPlugin` (`apps/studio/vite.config.ts`) with action kinds: `setup-infra`, `teardown-infra`, `set-secret-value`, `secret-status`, `stop`, `health`
- update `resolveDeploymentActionFromSettings` (`packages/plugins/src/deployment/actions.ts`) to return supported descriptors for `stop` and `health` for Cloud Run
- `stop` for Cloud Run resolves to `gcloud run services delete <service-name> --quiet --project=<projectId> --region=<region>`; idempotent (succeeds with a clear "already deleted" message if the service doesn't exist)
- `teardown-infra` is implemented as a sequenced wrapper: first runs `gcloud run services delete --quiet` for every Cloud Run service the deployment plan declares (each tolerating "not found"), then runs `terraform destroy -auto-approve` in the terraform directory. The ordering is required because terraform does not own the Cloud Run service resource but its IAM/WIF bindings are referenced by it; destroying those bindings while the service is still up causes dependency errors
- `health` performs an actual HTTP probe and returns the result. The middleware does `fetch(healthUrl, { signal: AbortSignal.timeout(5000), redirect: "manual" })` and returns `{ ok, statusCode, durationMs, bodyPreview (first 512 bytes, utf-8), error? }`. No retries — one shot. Body is truncated because gateway responses can be arbitrarily large and we only want a smoke signal. Redirects are not followed because health probes should hit `/healthz` directly; any redirect is misconfiguration we want surfaced. Result is serialized into the existing action descriptor's `stdout` field so the Studio UI can render it the same way it renders other action results
- `health` for **Cloud Run** resolves the service URL once via `gcloud run services describe <service-name> --format='value(status.url)'`, then probes `<url>/healthz`. If `describe` returns no URL (service not deployed), report `supported: false` with reason "Service is not deployed" — do not attempt the probe
- `health` for **local** uses the already-known `http://localhost:<gatewayHostPortBase>/healthz`; no resolution step. If the gateway container is down, the probe fails with `ECONNREFUSED` and the error message surfaces directly — no special handling needed. Replaces the current "return healthUrl only" behavior so both targets behave symmetrically
- `set-secret-value` handler passes the value via stdin to `gcloud secrets versions add`; explicit log suppression for request/response on that endpoint; reject the request if `value` length is zero or absurdly large
- enforce that `setup-infra` and `teardown-infra` only run when a `terraform` binary is on PATH; emit a clear error otherwise
- add a `cut-major-version` action kind: invokes the `cutNewMajorVersion(gameRoot)` helper described in the Domain section. Pre-flight is enforced server-side too (not only in the UI's disabled state): refuse with a structured error if `git` is not on PATH, working tree is dirty, or the would-be tag already exists. On success return `{ previousMajorVersion, newMajorVersion, tagCreated, commitSha }` so the Studio can render confirmation and refresh the SugarDeploy section without a manual reload

## Verification

This epic is complete when, on a fresh GCP project with `gcloud auth application-default login` and `terraform` available:

1. A user creates a new game in the Studio, fills in the Cloud Run section's fields, saves the project, and the `deployment/google-cloud-run/terraform/` directory contains valid generated terraform with the GENERATED header and template-version stamp.
2. Clicking Setup Infra in the SugarDeploy workspace runs `terraform apply` successfully and produces Artifact Registry, runtime SA, IAM bindings, WIF pool/provider, and empty Secret Manager containers.
3. Clicking Set Value on each Secret Manager secret successfully creates a new version with the provided value, and the secret value never appears in any Studio log, browser console, or shell history.
4. Clicking Deploy builds the gateway image, pushes it to Artifact Registry, and deploys the Cloud Run service with `--set-secrets` references resolving correctly at startup.
5. The deployed service responds to the gateway's `/healthz` endpoint and accepts SugarAgent calls from a separately-deployed (or local) Studio. Clicking Health in the SugarDeploy workspace returns a real probe result (`ok: true`, `statusCode: 200`, `durationMs`, truncated body) — not just a URL to copy.
6. Clicking Stop removes only the Cloud Run service (`gcloud run services delete`); Artifact Registry, IAM, WIF, and Secret Manager state remain. A subsequent Deploy brings the service back without re-running Setup Infra.
7. Clicking Teardown Infra deletes any remaining Cloud Run service(s) first, then runs `terraform destroy` cleanly without dependency errors, removing everything the plugin created.
8. Deleting the GCP project entirely (`gcloud projects delete`) removes all remaining state with no manual cleanup required.
9. With v1 deployed and running, clicking **Cut New Major Version** with a clean working tree creates the `v1.0.0` git tag, bumps the project metadata to `majorVersion: 2`, and commits the bump. The SugarDeploy section's default `gcp_project_id` now resolves to `${gameSlug}-v2`. Running Setup Infra + Deploy from this state stands up v2 in a fresh GCP project; v1 continues running in its original GCP project untouched. Checking out the `v1.0.0` tag in a `git worktree`, opening that worktree in Studio, and confirming SugarDeploy resolves back to the v1 GCP project demonstrates the patch-an-old-major path works.

## Stories

Stories are ordered for solo-dev linear execution. The Deliverables section above is the master detail; per-story descriptions stay light and lean on Deliverables by topic rather than restating them. Each story has an explicit **Exit** criterion that leaves the codebase in a working, verifiable state.

Dependency shape: `45.1 → 45.2 → 45.3 → 45.4 → 45.5 → 45.6 → 45.9`, with `45.7` parallel-after `45.2` and `45.8` parallel-after `45.1`.

### 45.1 — Project-versioning foundation

Adds `majorVersion: number` to the top-level game project metadata type in `packages/domain/src/project/`, defaulting to `1` for projects that lack the field (zero-effort migration for in-flight projects like wordlark). Surfaces it on `GameProject` through the existing project-loading path so the deployment plugin and Studio can read it without a special-case fetch. Updates the deployment plugin's default `gcp_project_id` and `service_name_prefix` derivation to `${gameSlug}-v${majorVersion}` — override fields continue to win when present. Updates the existing `deploy.sh` and `service.yaml` generators so the resolved names flow through the same defaulting path, avoiding inconsistency across files emitted in different stories.

Also surfaces the project identity globally in the Studio header — because `majorVersion` is a project-level property, not a deployment-specific one, it belongs in a project-level UI affordance and not buried inside the SugarDeploy section. The Studio header's bare "📁 Game" menu button is replaced (when a project is loaded) with the project's `displayName` plus a small `v${majorVersion}` `Badge` next to the 📁 menu trigger, keeping the menu behaviour unchanged. When no project is loaded, only the "Sugarmagic" brand is shown.

**Exit:** opening wordlark in the Studio loads it with `majorVersion: 1`; saving with no overrides set regenerates the existing `deploy.sh` and `service.yaml` with version-namespaced names — the resolved `gcp_project_id` and `service_name_prefix` are both `wordlark-v1`, and the final Cloud Run service name (prefix + service-unit id) carries `wordlark-v1-` as its prefix. The Studio header shows `📁 Wordlark  v1` when wordlark is the active project. `resolveSecretManagerName(plan, "anthropic-api-key")` will return `wordlark-v1-anthropic-api-key` once 45.5 lands (the helper itself is implemented in 45.5; the version-derived prefix is locked in here).

### 45.2 — Cloud Run terraform generators

Implements the four terraform generators (`buildCloudRunTerraformMainFile`, `buildCloudRunTerraformVariablesFile`, `buildCloudRunTerraformOutputsFile`, `buildCloudRunTerraformTfvarsFile`) and wires them into `buildGoogleCloudRunManagedFiles` so saving a project emits the files under `deployment/google-cloud-run/terraform/`. Adds the template-version stamping helper, the (empty) `TERRAFORM_RENAME_LEDGER`, and `readTemplateVersionFromDisk`. Adds `.gitignore` entries for `terraform.tfstate` and `.terraform/`. No Studio UI changes — the generators consume whatever overrides exist plus the 45.1 defaults.

**Exit:** saving wordlark produces a populated `deployment/google-cloud-run/terraform/` directory; `cd` into it + `terraform init` + `terraform validate` succeeds against a real GCP project (no `apply` yet); the generated `main.tf` carries both the `GENERATED BY SUGARMAGIC` header and a `# SUGARMAGIC TEMPLATE VERSION: 1` stamp; `terraform.tfstate` and `.terraform/` are gitignored.

### 45.3 — Studio fields + shared-project warning banner

Extends the SugarDeploy Cloud Run section UI with input fields for `gcp_project_id`, `region`, `github_repo` (with paste-forgiveness for `https://github.com/owner/repo` and `git@github.com:owner/repo.git` forms, validation against `^[A-Za-z0-9._-]+/[A-Za-z0-9._-]+$`), `service_name_prefix`, and `runtime_service_account_name`. Adds the non-blocking shared-project warning banner that fires when another open game's `gcp_project_id` matches the current game's. Pure data-binding + visual affordance — no new host actions. The version display is a global header element (landed in 45.1), so the SugarDeploy section does not duplicate it.

**Exit:** typing in each field persists to `gameProject.deployment.targetOverrides["google-cloud-run"]` and the next save reflects the override in the generated tfvars; opening two test games with the same project id renders the warning banner in both; clearing the override returns to the `${gameSlug}-v${majorVersion}` default.

### 45.4 — Setup Infra + Teardown Infra lifecycle

## Resolved Decisions

- **`terraform.tfvars` is fully managed.** Plugin emits it with the `GENERATED BY SUGARMAGIC` header on every save; hand-edits get overwritten. Rationale: the Studio is the single input surface for all terraform variables. An unmanaged `terraform.tfvars` (the common terraform convention) is a tempting hiding spot for API keys — leaving it unmanaged would create a hole that bypasses the `setSecretValue` host action's no-log discipline and routes secrets through `terraform.tfstate` instead of Secret Manager. Fully managed closes that hole.

- **Template-version drift handling is uniform regardless of gap size.** The plugin always emits the correct historical `moved {}` chain (walked from the on-disk version up to current via the `TERRAFORM_RENAME_LEDGER`), and the Studio shows a non-blocking drift banner when on-disk version < current. There is no separate "N-2 or older" branch — N-1 and N-5 take the same path. Rationale: "refuse to operate" is incoherent when the plugin *is* the migration tool; "silent fix-up" can mask state drift in the solo-dev / local-state setup this epic targets. Notify + proceed gives the user something to verify (`terraform plan`) and matches AGENTS.md's "goals must be verifiable" rule. The ledger retains entries indefinitely for v1; a future plan may cap retention once the version count makes generated terraform unwieldy.

- **Cross-game GCP-project enforcement is docs + a non-blocking Studio warning, not plan-layer validation.** The plugin-generated `deployment/google-cloud-run/README.md` and the SugarDeploy workspace copy state explicitly that one GCP project per game per environment is the supported model. The Studio additionally surfaces a warning banner whenever the current game's `gcp_project_id` matches another open game's. Neither path blocks save or any action. Rationale: plan-layer validation requires the plugin to know about every other game's deployment overrides at save time — cross-game state-awareness in a plugin that's otherwise game-scoped. The actual failure mode (`terraform destroy` from game A nuking IAM game B depends on; `gcloud projects delete` killing both at once) is solo-dev foot-gun territory, not a multi-tenant safety requirement. Notify + document + let the user proceed is the right intensity, and it mirrors the template-drift banner pattern above.

- **Game-project versioning is `majorVersion` metadata + git tags + version-namespaced GCP projects, with a first-class Cut New Major Version button.** One game directory per game forever; git is the source of truth for what code is at what version; the metadata field tells the deployment plugin which version slot the current working tree represents; the plugin derives the default `gcp_project_id` and `service_name_prefix` from `${gameSlug}-v${majorVersion}` so coexisting major versions land in isolated cloud projects automatically. The Studio's Cut New Major Version button shells the tag + bump + commit sequence so the operation is one click, not a checklist. Rationale: the directory-suffix-as-version approach (`wordlark/`, `wordlark-v2/`, etc.) is a copy-the-whole-project crutch that duplicates source, makes cross-version bug fixes a cherry-pick chore, and breaks the "one source of truth" rule from AGENTS.md. Git already solves the "what code is at this version" problem; the metadata mirror exists only so the deploy plugin can target the right cloud slot without round-tripping through `git describe`. Minor/patch versioning is intentionally out of scope for v1 — the only version distinction that affects deploy slotting is the major.
